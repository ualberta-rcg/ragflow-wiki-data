Vous trouverez ici l'information sur la migration d'instances virtuelles (ou VM pour ''virtual machines'') à partir du nuage Arbutus de la précédente génération vers le nuage Arbutus amélioré de la nouvelle génération. Puisque vous connaissez bien votre travail, nous vous recommandons de gérer vous-mêmes la migration vers le nouveau nuage selon les logiciels que vous utilisez et selon votre disponibilité.   

Il est important de migrer toutes vos ressources infonuagiques (instances, volumes de stockage, conteneurs de stockage objet, réseaux, clés, etc.) vers le nouveau nuage Arbutus puisque le système précédent deviendra hors service au cours de 2026. <b>'''La date limite pour migrer vos ressources est le 31 août 2026</b>. Ceci s'applique autant aux ressources allouées par concours qu'à celles qui ont été obtenues par le service d'accès rapide.''' 

Nous décrivons ici quelques méthodes de migration. Avec votre équipe de recherche, vous devez identifier la ou les approches en fonction de votre projet.

Cette information peut susciter des questions ou encore, vous voudrez peut-être discuter avec notre équipe technique de la meilleure approche dans votre cas particulier. Vous pouvez alors écrire à [mailto:cloud@tech.alliancecan.ca cloud@tech.alliancecan.ca].

<span id="Planning_your_cloud_migration"></span>
= Planifier la migration =

Les questions suivantes sont en rapport avec les ressources qui se trouvent sur le nuage Arbutus de la précédente génération. Elles vous aideront à planifier leur migration. 

* Quelles sont les ressources qui doivent être migrées vers le nouveau nuage Arbutus? Il n'est pas obligatoire de migrer toutes les ressources; par exemple, des volumes ou des instances qui ne sont plus nécessaires peuvent être supprimés. '''Créez une liste de toutes les ressources qui doivent être migrées.'''
* Vos instances sont-elles éphémères ou sont-elles basées sur des volumes? Les instances basées sur des volumes démarrent à partir d'un volume (par exemple, /dev/vda) et peuvent également avoir d'autres volumes (par exemple, /dev/vdb). Les instances éphémères ne démarrent pas à partir d'un volume. '''Listez les instances basées sur des volumes séparément des instances éphémères.'''
* Vos volumes font-ils plus de 150Go? Si c'est le cas, ils doivent être migrés à l'aide de Globus. '''Identifiez les volumes de plus de 150Go.'''
* Avez-vous utilisé un système de déploiement automatisé (par exemple, Terraform ou Ansible) sur le nuage de la précédente génération? '''Si c'est le cas, les mêmes outils d'automatisation doivent être utilisés pour la migration.'''
* Utilisez-vous des entrées DNS personnalisées? Ces entrées devront être mises à jour, car le nouveau nuage Arbutus utilise des plages d’adresses IP flottantes différentes de celles du système précédent.
* Pour gérer vos ressources infonuagiques, utilisez-vous le tableau de bord OpenStack (interface web Horizon) ou l’interface en ligne de commande (CLI) OpenStack? '''Les migrations simples peuvent être effectuées via l’interface web; par contre, les migrations plus complexes peuvent nécessiter un accès à l'interface en ligne de commande.'''
* Les membres de votre équipe ont-ils tous un compte actif avec l'Alliance? Veuillez noter que le partage de compte est strictement interdit. '''Toute personne ayant besoin d’un compte doit en [[Apply for a CCDB account/fr|faire la demande sur CCDB]].'''
* Comment gérerez-vous les interruptions de service requises pour la migration? Selon l’étendue de la migration, '''l’interruption peut durer de quelques heures à quelques jours. Qui devez-vous informer?'''
* Vos ressources ont-elles été allouées via le service d'accès rapide? Si c'est le cas, '''vous devrez soumettre une demande de migration à [mailto:cloud@tech.alliancecan.ca cloud@tech.alliancecan.ca].'''

Avec les réponses à ces questions, vous pourrez élaborer un plan de migration.

= Préparer l'environnement de migration =

# '''IMPORTANT''' : Avant de commencer, faites une copie de sauvegarde des données importantes. Le nuage est doté de systèmes de stockage redondants, mais nous ne faisons pas de sauvegarde des instances. '''La personne responsable du projet doit copier les données qui seront migrées.'''
# Connectez-vous aux deux nuages avec les informations d'identification pour votre compte et téléchargez les fichiers RC dans ''Project -> API Access -> Download OpenStack RC File''. Ces fichiers servent à configurer les variables d'environnement utilisées par les outils ligne de commande OpenStack.
# Copiez les fichiers RC d'OpenStack sur l'hôte que vous utiliserez pour la migration et suivez les instructions de la section ''Modifier le fichier RC du nouveau Arbutus'' ci-dessous.
# Testez le ou les fichiers RC pour vérifier que vous avez accès à vos projets dans les deux nuages.
## Activez un fichier RC en l'exécutant (<code>source opensrc.sh</code>) dans une session shell. Un seul fichier RC à la fois peut être actif dans une session shell.
## Testez votre configuration en exécutant <code>openstack volume list</code>.
# Migrez les clés SSH :
## Depuis le tableau du nuage Arbutus précédent, naviguez vers ''Compute -> Key Pairs''. Cliquez sur le nom de la paire de clés que vous voulez et copiez la valeur de la clé publique.
## Depuis le tableau de bord Horizon du nouveau nuage Arbutus, naviguez vers ''Compute -> Key Pairs''. Cliquez sur ''Import Public Key'', nommez votre paire de clés et collez-la dans le champ pour la clé publique du nuage précédent.
## Votre paire de clés devrait maintenant être importée dans le nouveau nuage. Répétez les étapes ci-dessus pour chaque paire de clés nécessaire.
## Vous pouvez également générer de nouvelles paires de clés ou les importer avec la commande <code>openstack keypair create --public-key <public-keyfile> <name></code>.
# Migrez les groupes de sécurité et les règles.
## Sur le nuage précédent, naviguez vers ''Network -> Security Groups'' et notez les groupes de sécurité existants ainsi que leurs règles associées.
## Sur le nouveau nuage, naviguez vers ''Network -> Security Groups'' et recréez les groupes de sécurité et leurs règles associées, au besoin.
## Ne supprimez aucune des règles de sécurité de Egress IPv4 et IPv6 créées par défaut. La suppression de ces règles peut empêcher vos instances de récupérer les données de configuration du service de métadonnées OpenStack en plus d’entraîner plusieurs autres problèmes.
## Les règles et les groupes de sécurité peuvent également être créés via l'interface de ligne de commande (CLI). Notre exemple concerne uniquement le port HTTP 80. Modifiez-le selon vos besoins.<br> <code>openstack security group create <nom-du-groupe></code> <br><code>openstack security group rule create --proto tcp --remote-ip 0.0.0.0/0 --dst-port 80 <nom-du-groupe></code>
## Pour afficher les règles via l'interface de ligne de commande (CLI),
##* ​​exécutez <code>openstack security group list</code> pour lister les groupes de sécurité disponibles;
##* ​​exécutez <code>openstack security group rule list</code> pour afficher les règles du groupe.
# Planifiez une interruption de service. En général, l'arrêt des services suivi de la fermeture de l'instance est la meilleure façon d'éviter la corruption ou l'incohérence des données après la migration. Les petits volumes peuvent être copiés assez rapidement; par exemple, un volume de 10Go sera copié en moins de cinq minutes, mais les volumes plus importants (par exemple, 100Go) peuvent prendre de 30 à 40 minutes. Tenez-en bien compte. De plus, les adresses IP flottantes seront différentes; assurez-vous donc que la durée de vie (TTL) de vos enregistrements DNS possède une valeur faible afin que les modifications se propagent le plus rapidement possible.

= Modifier le fichier RC du nouveau Arbutus =
Après avoir téléchargé le fichier RC du nouveau nuage, modifiez-le en ajoutant les lignes suivantes&nbsp;:

<source>
export OS_AUTH_TYPE=v3websso
export OS_IDENTITY_PROVIDER=atmosphere
export OS_PROTOCOL=openid
export OS_PROJECT_DOMAIN_NAME=default
</source>

Supprimez les lignes qui contiennent 

<source>
export OS_USER_DOMAIN_NAME="atmosphere" 
if [ -z "$OS_USER_DOMAIN_NAME" ]; then unset OS_USER_DOMAIN_NAME; fi
</source>

Supprimez aussi

<source>
echo "Please enter your OpenStack Password for project $OS_PROJECT_NAME as user $OS_USERNAME: " 
read -sr OS_PASSWORD_INPUT 
export OS_PASSWORD=$OS_PASSWORD_INPUT </code>
</source>

Le fichier devrait maintenant contenir des lignes semblables à ceci&nbsp;:  
<source>
export OS_AUTH_URL=https://identity.arbutus.alliancecan.ca/ 
export OS_PROJECT_ID=xIDx 
export OS_PROJECT_NAME=" xIDx " 
export OS_PROJECT_DOMAIN_ID=" xIDx " 
unset OS_TENANT_ID 
unset OS_TENANT_NAME 
export OS_USERNAME=" xIDx " 
export OS_REGION_NAME="RegionOne" 
export OS_INTERFACE=public 
export OS_IDENTITY_API_VERSION=3 
export OS_AUTH_TYPE=v3websso  
export OS_IDENTITY_PROVIDER=atmosphere  
export OS_PROTOCOL=openid 
export OS_PROJECT_DOMAIN_NAME=default 
</source>

Créez ensuite un environnement virtuel pour installer le client OpenStack et les autres logiciels dont vous avez besoin. 
<source>
$ python3 -m venv openstack 
$ source openstack/bin/activate 
$ pip install python-openstackclient  keystoneauth-websso python-manilaclient 
</source>

<span id="Manual_or_orchestrated_migration"></span>
== Migration manuelle ou orchestrée ==

Ici, les nouvelles instances et les nouveaux volumes sont créés sur le nouveau nuage Arbutus, avec les mêmes caractéristiques que sur le nuage Arbutus précédent. Les fichiers et données qui sont nécessaires sont copiés. Règle générale, les grandes lignes de la procédure sont ː 

# Si vous utilisez des images personnalisées, copiez les images Glance du nuage Arbutus précédent sur le nouveau nuage ou simplement commencer avec une nouvelle image de base sur le nouveau nuage. 
# Installez et configurez les services sur la ou les instances.
# Copiez les données des anciennes instances vers les nouvelles instances (voir [[#Méthodes_de_copie_de_données|Méthodes de copie de données]] ci-dessous).
# Assignez des adresses IP flottantes aux nouvelles instances et faites la mise à jour des DNS.
# Mettez fin aux anciennes instances et supprimez les anciens volumes.

Ces étapes peuvent être effectuées manuellement ou être orchestrées avec 
[https://docs.ansible.com/ansible/2.5/modules/list_of_cloud_modules.html Ansible], [https://www.terraform.io/docs/providers/openstack/ Terraform] ou [https://wiki.openstack.org/wiki/Heat Heat].
Le présent document ne traite pas de ces outils; cependant, si vous les utilisez sur le nuage précédent, ils devraient fonctionner de la même manière sur le nouveau nuage.

<span id="Migrating_volume-backed_instances"></span>
== Migration d'instances associées à des volumes ==

Comme leur nom l’indique, chacune de ces instances est associée à un volume persistant qui contient le système d’exploitation et les données nécessaires. Une bonne pratique est de créer des volumes distincts pour le système d’exploitation et pour les données.

<span id="Migration_using_Glance_images"></span>
=== Migration avec des images Glance ===

Ce scénario est recommandé dans le cas de volumes de moins de 150Go. Pour les volumes plus grands, la [[Arbutus Migration Guide/fr#Migration_manuelle_ou_orchestrée|migration manuelle ou orchestrée]] est préférable.

# Ouvrez deux sessions SSH pour l'instance que vous prévoyez migrer.
# Dans une session, chargez le fichier RC d'OpenStack pour le nuage précédent. Dans l'autre session, chargez le fichier RC d'OpenStack pour le nouveau nuage.<br> L'utilisation de la commande <code>screen</code> est recommandée pour parer à une déconnexion SSH. Pour installer le paquet screen, exécutez <code>dnf install screen</code>.
# Dans l'instance du nuage précédent, installez l'interface de ligne de commande OpenStack dans un shell racine (root).
<source>
dnf install epel-release
dnf install python-devel python-pip gcc
pip install python-openstackclient
</source>
# <li value=4> Dans l'interface web du nuage précédent, fermez l'instance et détachez le volume.<br> Si le volume sert au démarrage d'une instance, supprimez l'instance, mais conservez le volume.<br>Créez une image du volume (''Volumes -> Volumes'' et ''Upload to Image'' du menu déroulant).<br>Assurez-vous de sélectionner RAW comme format de disque.
# <li value=5> Sur la ligne de commande, vous pouvez aussi entrer
<code>openstack image create --volume <nom/ID du volume> <newimagename> --private</code><br>
Cette commande s'exécute en arrière-plan et peut prendre un certain temps. Une fois l'image créée, elle apparaît sous ''Compute -> Images'' avec le nom que vous avez spécifié à l'étape précédente. Vous pouvez obtenir l'identifiant de l'image en cliquant sur son nom. L'état de la ligne de commande passera progressivement de ''saving'' à ''active''; cela peut prendre une heure ou plus, selon la taille de votre volume.
<code>openstack image show <newimagename></code>
# <li value=6> Dans la session du nuage précédent, téléchargez l'image (remplacez <filename> et <image-id> par leurs valeurs réelles).
<code>openstack image save --file <filename> <image-id></code>
# <li value=7> Dans la session du nouveau nuage, sur l'hôte de migration, chargez l'image (remplacez <filename> par le nom dans l'étape précédente; <image-name> peut être n'importe quel nom).
<code>openstack --os-cloud <cloud> image create --private --file <file> <newImageName></code>
# <li value=8> Vous pouvez maintenant créer un volume à partir de l'image chargée. Dans l'interface web du nouveau nuage, naviguez vers ''Compute -> Images''. L'image chargée à l'étape précédente devrait s'y trouver. Dans le menu déroulant pour l'image, sélectionnez l'option ''Create Volume'' pour créer le volume. Ce volume peut ensuite être attaché à des instances ou utilisé pour démarrer une nouvelle instance.
# <li value=9> Une fois vos instances et volumes migrés et validés, et une fois tous les enregistrements DNS associés mis à jour, veuillez supprimer vos instances et volumes sur le nuage précédent. 

<span id="Alternative_method:_Migrating_a_volume-backed_instance_using_Linux_&#039;dd&#039;"></span>
=== Autre option avec Linux 'dd' ===

# Lancez une instance sur le nuage précédent avec la plus petite configuration possible (p1-1.5gb). Appelons-la ''hôte de migration temporaire''. Les instructions ci-dessous supposent que vous utilisez AlmaLinux 9, mais toute distribution Linux disposant de Python et de Pip devrait convenir.
# Connectez-vous à l'instance via SSH et installez l'interface de ligne de commande OpenStack (CLI) dans un shell racine (root).
<source>
dnf install epel-release
dnf install python-devel python-pip gcc
pip install python-openstackclient
</source>
# <li value=3> Pour vérifier l'installation, essayez d'exécuter OpenStack en ligne de commande. Pour plus d'information, notamment sur l'installation de la CLI OpenStack sur des systèmes autres qu'AlmaLinux, consultez https://docs.openstack.org/newton/user-guide/common/cli-install-openstack-command-line-clients.html. 
# Copiez votre fichier RC OpenStack du nouveau nuage vers l'hôte de migration temporaire et exécutez-le. Vérifiez que vous pouvez vous connecter à l'API OpenStack sur le nouveau nuage avec la commande <code>openstack image list</code>.
# Supprimez l'instance à déplacer, mais NE SUPPRIMEZ PAS le volume auquel elle est attachée. Ce volume peut désormais être attaché à l'hôte de migration temporaire que nous avons créé.
# Dans l'interface web du nuage précédent, naviguez vers ''Volumes -> Volumes''. Dans le menu déroulant, sélectionnez ''Manage Attachments'' et attachez le volume à l'hôte de migration temporaire. Notez le périphérique auquel le volume est attaché (généralement /dev/vdb or /dev/vdc).
# À l'aide de l'utilitaire <code>dd</code>, créez une image à partir du disque identifié à l'étape précédente. Dans l'exemple suivant, nous l'avons nommée ''volumemigrate''. Une fois la commande terminée, vous recevrez un résultat affichant les détails de l'image créée.
 dd if=/dev/vdb | openstack image create --private --container-format bare --disk-format raw "volumemigrate" 
# <li value=8> Dans l’interface web du nouveau nuage, naviguez vers ''Compute -> Images''. L'image peut désormais être utilisée pour lancer des instances. Si vous souhaitez que les données soient persistantes, assurez-vous de créer un nouveau volume lors du lancement de l'instance.

# <li value=9> Une fois vos instances et volumes migrés et validés, et les enregistrements DNS associés mis à jour, veuillez supprimer vos instances et volumes sur le nuage précédent. 

=== Migration de grands volumes avec Linux 'dd' ===
Dans le cas de volumes, l'utilisation des images n'est pas recommandée. Copiez plutôt vos données dans de nouveaux volumes avec rsync ou un autre outil de copie, lorsque possible. Si ce n'est pas possible, par exemple dans le cas d'un volume amorçable (''bootable''), vous pouvez utiliser la commande <code>dd</code> pour produire une copie identique sur le nouveau nuage.

Sauvegardez les données importantes avant d'exécuter cette procédure.

# Créez une instance temporaire sur le nuage précédent (1 à 1,5 Go devraient convenir) et faites de même sur le nouveau cloud.
# Attribuez les deux adresses IP flottantes ci-dessus auxquelles vous pouvez vous connecter avec SSH.
# Installez les paquets suivants sur l'instance temporaire du nuage précédent.
<source>
dnf install epel-release
dnf install pv
dnf install screen
</source>
# <li value=4> Sur la nouvelle instance temporaire, exécutez : <code>chmod u+s /bin/dd</code>
# Copiez la clé privée SSH que vous utilisez pour vous connecter en tant qu'utilisateur sur la nouvelle instance temporaire vers l'instance temporaire du nuage précédent.
# Assurez-vous que les règles de sécurité SSH autorisent l'instance temporaire du nuage précédent à se connecter avec SSH à la nouvelle instance temporaire.
# Pour chaque volume que vous souhaitez déplacer du nuage précédent vers le nouveau nuage,
## créez un volume vide de même taille sur le nouveau cloud et marquez-le comme amorçable s'il s'agit d'un volume d'amorçage;
## attachez le volume ci-dessus à l'instance temporaire sur le nouveau cloud;
## attachez le volume que vous souhaitez copier du nuage précédent à l'instance temporaire. Remarque&nbsp;: vous devrez peut-être supprimer l'instance à laquelle il est actuellement attaché. NE SUPPRIMEZ PAS le volume.
# Sur l'instance temporaire, exécutez les commandes ci-dessous. Ceci suppose que le volume source sur le nuage précédent est attaché à l'instance temporaire sous le nom /dev/vdb, que sa taille est de 96Go, que la clé SSH utilisée pour se connecter à l'instance temporaire est key.pem et que le volume de destination sur Arbutus  est attaché à l'instance temporaire sous le nom /dev/vdb. Remplacez également ''xxx.xx.xx.xx'' par l'adresse IP réelle de l'instance Arbutus à laquelle vous vous connecterez. La commande <code>screen</code> est utilisée pour parer à une déconnexion de votre session SSH.
<source>
screen

Dans le nouveau nuage, vous disposez maintenant d'une copie identique du volume du nuage précédent et vous pouvez l'utiliser pour y lancer des instances.

== Petits volumes de données : rsync + ssh ==
Pour les plus petits volumes, rsync+ssh offre de bonnes vitesses de transfert et, comme Globus, travaille de manière incrémentale. Un cas type serait :  

Voici un exemple de cas type&nbsp;:

# Connectez-vous avec SSH à l’instance sur le nuage Arbutus précédent, qui possède le volume principal. Prenez note du chemin absolu que vous voulez copier dans l’instance sur le nouveau nuage.
# Lancez rsync sur SSH. Dans l’exemple suivant, on suppose qu’il existe une connexion sans mot de passe via des [[SSH Keys/fr|clés SSH]]. Utilisez le code suivant avec les valeurs réelles. 
#: <code> rsync -avzP -e 'ssh -i ~/.ssh/key.pem' /local/path/ remoteuser@remotehost:/path/to/files/ </code>
# Vérifiez que les données ont bien été copiées dans l’instance du nouveau nuage, puis supprimez les données sur le nuage précédent.

# updating firewall rules to use any new IP addresses and networks if a host-based firewall (e.g. iptables, firewalld, etc.) is used;

# altering usernames (e.g., 'root'@'192.168.65.%) in MySQL;

<span id="Migrating_the_CephFS_shared_filesystem"></span>
= Migration du système de fichiers partagé CephFS =

The CephFS shared filesystem on the new cloud is a distinct and separate service; any desired data must be intentionally migrated there. 

Management of legacy shares, including operations for creation, deletion, and key management, is controlled through the legacy Arbutus Cloud. However, once a legacy share and key are created, those resources can be accessed from a virtual machine in the new Arbutus cloud. Similarly, creation and management for shares in the new cloud is done exclusively in the new cloud environment. 

Both legacy shares and new shares can be mounted on virtual machines in the new Arbutus cloud. The following procedure is recommended to ensure data integrity when migrating data between legacy and new shares.
# For each share in the legacy cloud, create an equivalent share in the new cloud.
# Mount both shares to separate mount locations on the same virtual machine in the new cloud. 
# Use a data copy tool such as rsync to transfer the data from the old share to the new share. 

The procedure for mounting legacy shares can be found in [[CephFS|our CephFS wiki page]]. 

Creating the equivalent share in the new Arbutus cloud follows a similar procedure, with a few essential differences: 
# You must create the new share and access keys with the new cloud web user interface.
# You must create a separate ceph.conf file, with a distinct name such as ''ceph-new.conf''.
# The <code>mon_host</code> config value will need to be updated for the new share only, in the ''ceph-new.conf'' file:
## Legacy value: <code>“10.30.201.3:6789,10.30.202.3:6789,10.30.203.3:6789”</code>
## New value: <code>“[v2:134.87.15.61:3300/0,v1:134.87.15.61:6789/0] [v2:134.87.15.62:3300/0,v1:134.87.15.62:6789/0] [v2:134.87.15.63:3300/0,v1:134.87.15.63:6789/0]”</code>
# When mounting the new share, an extra value in the mount command is required after the <code>-o</code> to specify the new configuration file ''conf=/etc/ceph/ceph-new.conf''.

Once both shares are mounted, use rsync to transfer the data. The a, v, and P flags for rsync are recommended. 

<code>rsync -avp /mnt/old-share/ /mnt/new-share/</code>

Keep in mind that depending on the size of your share, this may take a long time. To keep the session alive in case of a dropped connection, use a tool such as screen or tmux. 

<span id="Migrating_object_storage"></span>
= Migration du stockage objet =

Object storage on the new cloud is a distinct and separate service; any desired data must be intentionally migrated there. 

Management for legacy buckets and objects, including operations for creation, deletion, object manipulation, and key management, are controlled through the legacy cloud. Similarly, creation and management for buckets and objects in the new Arbutus cloud is done exclusively in the new Arbutus cloud environment. 

Migrating data to the new cloud object storage can be done using a variety of methods and tools. If you are familiar with the options, feel free to use whichever method works best for your data. 
 
'''New Arbutus Cloud Object Storage Endpoint:''' https://object-arbutus.alliancecan.ca 

'''Legacy Arbutus Cloud Object Storage Endpoint:''' https://object-arbutus.cloud.computecanada.ca 

If you’re using bucket ACLs, check if the tool you use copies them correctly, otherwise you must recreate them in the new environment. Most tools do not preserve bucket ACLs. Keep in mind that if you reference any user or project UUIDs, they will be different in the new Arbutus cloud. 

Additionally, object storage in the new cloud uses tenants; therefore, bucket name collisions will only happen within an individual project instead of across all projects. When authenticating with either the Swift or S3 API, the tenant is inferred from the user/key provided. However, for public access to buckets with no authentication, the tenant must be specified. The Tenant ID is identical to the OpenStack Project ID. The URL for unauthenticated Swift access can be found via the Horizon interface, while the URL for unauthenticated S3 access is of the following format:

<code>https://object-arbutus.alliancecan.ca/<tenant-id>:<bucket-name>/<object-name></code>

Nous recommandons rclone; les ACL du bucket ne seront pas copiés et tous les accès copiés seront par défaut privés, comme dans cet exemple

# Install rclone with https://rclone.org/install/. 
# Create S3 credentials in both the legacy and the new Arbutus clouds; see [[Arbutus object storage|our Arbutus object storage wiki page]].  
# Create a config file for rclone: 
## File location on linux/MacOS: <code>~/.config/rclone/rclone.conf</code>
## File contents, inserting your access and secret values for each environment: <source lang="text">[new] 
type = s3 
access_key_id = <RENEWAL ACCESS KEY> 
secret_access_key = <RENEWAL SECRET KEY> 
endpoint = https://object-arbutus.alliancecan.ca 
[legacy] 
type = s3 
access_key_id = <LEGACY ACCESS KEY> 
secret_access_key = <LEGACY SECRET KEY>
endpoint= https://object-arbutus.cloud.computecanada.ca</source>
#<li value="4"> Sync all buckets with <code>rclone sync legacy: renewal:</code>.
